В данном примере реализована некая полуабстрактная банковская архитектура, согласно заданию.
Объеденил архитектурное решение с некими ограничениями по входным значениям.
Некоторые моменты специально упрощены,чтобы не увеличения время выполнения:
- имена тестов
- CreditCardTests.test1() ожидаемое значение захардкожено чтобы не тратить время, в реальном проекте, конечно, была
бы реализована полная логика(неплохо с рандомными значениями) для проверки результатов фронта и/или бека
- не стал дублировать некоторые тесты, т.к. логика пополнения/снятия есть в базовом классе

Решение применять интерфейсы для пополнения/снятия было принято потому что, в депозитах не предусмотрено снятие.
Класс AbstractCurrencyBankProduct является базовым для всех валютных продуктов, т.к. именно они фигурируют в задании,
для создание нового продукта (например металлического счёта данная структура неподойдёт), решил не вводить ещё одну
абстракцию с одним полем name, чтобы не растягивать архитектуру и так же время (в целом, наверно он нужн будет для металлического счёта);
